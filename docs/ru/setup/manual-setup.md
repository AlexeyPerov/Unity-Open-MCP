# Ручная установка

Установите Unity Open MCP без Unity Hub Pro.

## Для кого это

Это путь **«сделай сам»**. Подойдёт, если вы:

- уже редактируете вручную `manifest.json` Unity и файлы конфигурации MCP-клиента, или
- предпочитаете точно контролировать, что именно устанавливается и фиксируется, или
- работаете на headless/CI-машине, где Hub GUI недоступен.

Хотите, чтобы установку выполнил ИИ-агент? См. [agent-setup.md](agent-setup.md)
(вставьте промпт из README в ваш ИИ-клиент).

Если вы никогда не открывали терминал и не редактировали JSON, путь через
**мастер** намного проще — см. [wizard-setup.md](wizard-setup.md). Он делает всё
ниже автоматически и поясняет каждый шаг.

## Как Unity Open MCP устроен (две половины)

Unity Open MCP состоит из двух независимых половин, общающихся по петлевому
(loopback) сетевому интерфейсу вашей машины:

| Половина | Где находится | Откуда устанавливается | Что делает |
|---|---|---|---|
| **Сторона ИИ** | небольшая Node-программа (MCP-сервер) | **npm** (`npx`) | Ваш ИИ-клиент (Cursor, Claude, …) её запускает; она открывает инструменты MCP. |
| **Сторона Unity** | два пакета редактора Unity (bridge + verify) | **Unity Package Manager** (Git URL) | Работает внутри Unity и выполняет каждый вызов инструмента. |

Нужны **обе**. Сторона ИИ никогда не касается Unity напрямую — она обращается к
стороне Unity по HTTP. Ниже шаги устанавливают каждую половину по очереди.

Для работы над самими пакетами (локальный чекаут, сборка MCP-сервера, рабочие
процессы контрибьютора и мейнтейнера) см. [development-setup.md](development-setup.md).

## Требования

- **Unity 2022.3 LTS или новее**.
- **Node.js 18 или новее** — это нужно *только* потому, что MCP-сервер — небольшая
  Node-программа, которую ваш ИИ-клиент запускает в фоне. Вы **не** будете писать
  JavaScript или интерактивно его запускать.
  - Нет Node? Установите с <https://nodejs.org/> (кнопка **LTS** подойдёт). После
    установки перезапустите терминал, чтобы команды `node`/`npx` оказались в PATH.
    Проверьте через `node --version`.
- **MCP-клиент, поддерживающий stdio MCP-серверы** — Cursor, Claude Desktop,
  Claude Code, OpenCode, ZCode, Cline, Codex, VS Code Copilot, Gemini CLI или любой
  совместимый MCP-клиент. Мастер Hub настраивает их все одним кликом; ниже — фрагменты
  для наиболее частых ручных форм.

## 1) Добавьте пакеты Unity (сторона Unity)

Откройте `Packages/manifest.json` в вашем проекте Unity (он в корне папки проекта,
например `MyGame/Packages/manifest.json`) и добавьте две записи в объект `dependencies`.

### Установка через Git (рекомендуется)

```json
{
  "dependencies": {
    "com.alexeyperov.unity-open-mcp-bridge": "https://github.com/AlexeyPerov/unity-open-mcp.git?path=packages/bridge#bridge-v0.7.0",
    "com.alexeyperov.unity-open-mcp-verify": "https://github.com/AlexeyPerov/unity-open-mcp.git?path=packages/verify#verify-v0.7.0"
  }
}
```

### Опциональные доменные зависимости Unity

Доменные инструменты идут в комплекте с мостом. Некоторым требуется
соответствующий пакет Unity перед компиляцией, а большинству — явная активация в
сессии. Канонический каталог зависимостей, таблица активации и примеры пакетов —
в [Расширениях](../../extensions.md).

### Опциональные зависимости (в редакторе)

После установки моста вы можете добавлять и удалять доменные зависимости Unity без
ручного редактирования манифеста. Откройте **Tools → Unity Open MCP Bridge →
Extensions** и используйте панель **Optional Unity dependencies**: одна строка на
домен показывает статус «установлено/отсутствует», с действием **Install…** /
**Remove…** в один клик для каждого UPM-пакета. Unity повторно импортирует манифест
и перекомпилирует; встроенные инструменты регистрируются (или перестают
компилироваться) при следующей перезагрузке домена. Модальное окно настроек проекта
Unity Hub Pro отражает это как панель статуса только для чтения.

## 2) Настройте MCP-клиент (сторона ИИ)

Теперь укажите вашему ИИ-клиенту на MCP-сервер. Используйте
[Конфигурацию MCP-клиента](client-configuration.md) для канонического
клиент-специфичного пути и оболочки JSON/TOML/CLI. Объедините описанную запись
`unity-open-mcp` в конфигурацию клиента, не заменяя посторонние настройки.

Что это значит:

- `npx -y unity-open-mcp@0.7.0` скачивает и запускает именно эту версию MCP-сервера
  из npm. `-y` принимает промпт первого запуска; фиксация версии держит сервер в
  синхроне с пакетами bridge и verify, у которых тот же номер. Для перехода на новый
  релиз поднимите версию здесь и в вашем `manifest.json` Unity одновременно — см.
  [Версионирование](../../versioning.md).
- **Первый запуск может занять 10–60 секунд** на скачивание пакета — это нормально,
  не зависание. Последующие запуски быстрые.
- Prefer one-time install? `npm install -g unity-open-mcp` помещает его в PATH,
  затем используйте `"command": "unity-open-mcp"` (без `npx`/`args`). Обновляете
  вручную через `npm update -g unity-open-mcp`.

> ⚠️ **`UNITY_PROJECT_PATH` — главная ловушка установки №1.** Это должен быть
> **абсолютный** путь к корню вашего проекта Unity (папке, содержащей `Assets/`,
> `Packages/` и `ProjectSettings/`). При отсутствии сервер немедленно завершает
> работу, а неверный путь означает, что ИИ может управлять другим проектом Unity,
> не тем, который у вас открыт.

Основные переменные окружения для проекта, порта и пакетного режима собраны в общем
клиентском справочнике. Полная матрица переменных окружения для стартовых диалогов
и политика безопасности — только в [Политике диалогов](../../dialog-policy.md).

## 3) Запустите Unity и проверьте

1. Откройте **тот же** проект Unity (на который указывает `UNITY_PROJECT_PATH`) в редакторе.
2. Дождитесь компиляции скриптов — следите за статус-баром в правом нижнем углу.
3. Перезапустите MCP-клиент (чтобы он перечитал конфигурацию из шага 2).
4. Простейшая проверка: в Unity откройте **Tools → Unity Open MCP Bridge** — окно
   должно показать статус **connected**. Если так — готово.
5. Предпочитаете терминал? Убедитесь, что мост достижим:

```bash
curl -s "http://127.0.0.1:<port>/ping"
```

Порт выводится автоматически из пути вашего проекта; если вы задали
`UNITY_OPEN_MCP_BRIDGE_PORT`, используйте его. Его также можно прочитать в окне
моста на шаге 4. Успешный ping возвращает JSON с `"connected": true`.

Наконец, попросите ваш ИИ-клиент запустить любой инструмент Unity Open MCP (например,
вывести список доступных возможностей) — если он отвечает данными Unity, обе
половины общаются.

## 4) Опциональный CLI (CI и автоматизация)

```bash
npx -y unity-open-mcp@0.7.0 wait-for-ready --project /path/to/MyGame
npx -y unity-open-mcp@0.7.0 run-tool unity_open_mcp_capabilities --project /path/to/MyGame --json
```

На автономных машинах настройте обработку стартовых модальных окон через
[Политику диалогов](../../dialog-policy.md) (`UNITY_OPEN_MCP_DIALOG_POLICY` и связанные env-переменные).

## Устранение неполадок

Для этого пути сначала убедитесь, что Unity открыт на том же абсолютном
`UNITY_PROJECT_PATH`, компиляция завершена, а MCP-клиент перезапущен. По
подключению, listener-у, модальным окнам и восстановлению нерабочего моста
следуйте полному руководству [Устранение неполадок](../../troubleshooting.md).
Политика модальных окон и детали macOS Accessibility — в
[Политике диалогов](../../dialog-policy.md).
