# Установка для разработки

Данная инструкция покрывает установку среды для работы с локальным чекаутом проекта. Обычная установка описана в
[Ручной установке](manual-setup.md) и [Установке агентом](agent-setup.md).

## Требования

- Unity 2022.3 LTS или новее
- Node.js 18 или новее
- MCP-клиент, поддерживающий stdio-серверы

## 1. Установка локальных пакетов Unity

Укажите в `Packages/manifest.json` целевого проекта Unity на чекаут:

```json
{
  "dependencies": {
    "com.alexeyperov.unity-open-mcp-bridge": "file:../../unity-open-mcp/packages/bridge",
    "com.alexeyperov.unity-open-mcp-verify": "file:../../unity-open-mcp/packages/verify"
  }
}
```

Скорректируйте относительный путь под вашу раскладку. Проект Unity внутри этого
монорепозитория обычно может использовать:

```json
{
  "dependencies": {
    "com.alexeyperov.unity-open-mcp-bridge": "file:../../packages/bridge",
    "com.alexeyperov.unity-open-mcp-verify": "file:../../packages/verify"
  }
}
```

Unity перекомпилирует исходники пакетов после refresh/focus. Локальные для пакета
требования валидации описаны в `AGENTS.md` и README каждого пакета.

## 2. Сборка и настройка MCP-сервера

```bash
cd mcp-server
npm install
npm run build
```

Используйте команду локального чекаута из
[Конфигурации MCP-клиента](client-configuration.md), сохраняя форму конфигурации
для вашего клиента:

```json
{
  "command": "node",
  "args": ["/absolute/path/to/unity-open-mcp/mcp-server/dist/index.js"],
  "env": {
    "UNITY_PROJECT_PATH": "/absolute/path/to/project"
  }
}
```

Политика стартовых диалогов и требования macOS Accessibility принадлежат
[Политике диалогов](../../dialog-policy.md).

## 3. Опциональные встроенные домены

Доменные инструменты поставляются внутри моста. Соответствующие зависимости Unity
включают компиляцию пакетно-управляемых доменов; активация сессии определяет,
будет ли большинство групп видно MCP-клиенту.

Используйте [Расширения](../../extensions.md) как канонический каталог зависимостей,
примеры манифеста и таблицу активации. Авторинг community-паков описан отдельным
контрактом в [Контрибьюции — расширения](../../contributing/extensions.md).

## 4. Запуск и проверка

1. Откройте целевой проект Unity.
2. Дождитесь компиляции скриптов.
3. Перезапустите MCP-клиент, чтобы он перечитал локальную команду.
4. Вызовите `unity_open_mcp_ping` или `unity_open_mcp_capabilities`.

Для восстановления моста, listener-а, компиляции и test-worker-а —
[Устранение неполадок для контрибьюторов](../../troubleshooting-contributors.md).

## Рабочий процесс с community-пакетами

`packages/extensions/` зарезервирован под отдельные community-доменные пакеты.
Поставляемые домены встроены в мост и не используют отдельные записи extension-пакетов.

Локальный community-пакет можно установить со своим UPM id:

```json
{
  "dependencies": {
    "com.example.my-mcp-ext": "file:../../my-mcp-ext"
  }
}
```

Следуйте [Контрибьюции — расширения](../../contributing/extensions.md) в части
шлюзов компиляции, регистрации инструментов, тестов и владения документацией.

## Релизы мейнтейнеров

npm-пакет публикуется из `mcp-server/`; bridge и verify потребляются через git-теги.
Unity Hub Pro выпускается независимо.

[Версионирование и релизы мейнтейнеров](../../contributing/versioning.md) —
канонический источник по:

- источникам версий и генерируемым целям;
- проверкам синхронизации и CI-дрейфа;
- командам повышения версии для общего трио и Hub;
- пространствам имён тегов и релизным процессам;
- владению заметками о релизе в GitHub Release.

Панель мейнтейнера Hub может запускать сборку/тесты, npm dry-run/publish и общий
скрипт синхронизации версий. Она использует существующую npm-аутентификацию
мейнтейнера и не создаёт коммитов или тегов.
