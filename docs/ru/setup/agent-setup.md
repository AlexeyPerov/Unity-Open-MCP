# Установка агентом

**Аудитория: ИИ-агенты.** Следуйте этой процедуре, чтобы установить Unity Open MCP
в проект Unity. Выполняйте каждый шаг агента самостоятельно. Останавливайтесь и
сообщайте человеку только когда блок **USER ACTION** предписывает это.

Люди: этот путь **экспериментальный**. Предпочтительнее
[Ручная установка](manual-setup.md) или [Установка через мастер](wizard-setup.md).
Чтобы всё же попробовать установку агентом, вставьте промпт из
[README](../../../README.ru.md#быстрая-установка) в ваш ИИ-клиент.

## Жёсткие правила (не пропускать)

1. **Никогда не выдумывайте номера версий.** Не вспоминайте `0.x.y` из обучающих
   данных или прошлых чатов. Единственная допустимая версия — та, что вы
   получили на [Шаге 0](#шаг-0--определите-версию-релиза).
2. **Загружайте эту процедуру заново** (или читайте из локального чекаута
   монорепозитория). Не импровизируйте альтернативную установку по памяти.
3. **Копируйте скил байт-в-байт** (`curl` / `cp`). Не переписывайте, не
   сокращайте, не расширяйте и не «улучшайте» `SKILL.md`. **Не** вызывайте
   `unity_open_mcp_generate_skill` во время установки.
4. **Пишите один клиентский путь скила** (обнаруженный клиент). Не размножайте
   файл по всем папкам клиентов, если человек не просит.

## Цель

Установить обе половины Unity Open MCP для **одного** проекта Unity:

| Половина | Что устанавливается |
|---|---|
| **Сторона Unity** | Пакеты bridge + verify в `Packages/manifest.json` (привязка через Git URL) |
| **Сторона ИИ** | Конфигурация MCP-клиента, запускающая `npx -y unity-open-mcp@<VERSION>` с `UNITY_PROJECT_PATH` |

Затем скопировать основной агентский скил (байт-в-байт), передать перезапуск
человеку и по возможности выполнить проверку, когда инструменты станут доступны.

**Не** устанавливайте опциональные доменные пакеты Unity (NavMesh, Input System, …),
если человек не просит об этом. Направьте его к [Расширениям](../../extensions.md).

## Чеклист предусловий

Перед любыми изменениями:

1. **Определите корень проекта Unity** — каталог, содержащий `Assets/`, `Packages/`
   и `ProjectSettings/`. При необходимости поднимайтесь вверх от cwd рабочего
   пространства. Если такой каталог не находится — **остановитесь** и спросите у
   человека абсолютный путь к проекту.
2. **Определите абсолютный путь проекта** — без завершающего слеша. Он станет
   `UNITY_PROJECT_PATH`.
3. **Проверьте Node.js** — выполните `node --version`. Требуется **18+**.
   - Если отсутствует или устарел → **USER ACTION:** установите Node LTS с
     <https://nodejs.org/>, перезапустите терминал / ИИ-клиент, чтобы `node`/`npx`
     появились в `PATH`, затем попросите человека перезапустить эту установку.
4. **Определите MCP-клиент** (авто-определение; спросите один раз, если неоднозначно):
   - Cursor → наличие `.cursor/` или характерные для Cursor env/UI-признаки
   - Claude Code → `claude` CLI / сессия Claude Code
   - Claude Desktop → пути конфигурации приложения Claude Desktop
   - VS Code Copilot → `.vscode/` + Copilot MCP
   - Visual Studio Copilot → `.vs/`
   - OpenCode → `opencode` / `opencode.json`
   - ZCode → `.zcode/` / `.agents/`
   - Codex → `.codex/`
   - Иначе спросите: «Какой ИИ-клиент мне настроить?»

## Шаг 0 — Определите версию релиза

Прочитайте привязку релиза **до** любых правок манифеста или MCP.

1. Загрузите (или прочитайте локально, если монорепозиторий открыт):

   `https://raw.githubusercontent.com/AlexeyPerov/Unity-Open-MCP/master/version.json`

   Локальный чекаут: `version.json` в корне репозитория.
2. Разберите JSON и установите `VERSION` из поля `version` (например,
   `"0.7.0"` → `VERSION=0.7.0`).
3. Опционально сверьте с опубликованным npm:

   ```bash
   npm view unity-open-mcp version
   ```

   Если latest на npm и `VERSION` различаются, **предпочитайте `VERSION` из
   `version.json`** (это документированная привязка для этой процедуры) и
   сообщите человеку.
4. Далее каждая привязка использует тот же `VERSION`:

   | Артефакт | Привязка |
   |---|---|
   | npm MCP-сервер | `unity-open-mcp@<VERSION>` |
   | Bridge UPM | `…/packages/bridge#bridge-v<VERSION>` |
   | Verify UPM | `…/packages/verify#verify-v<VERSION>` |

**Остановитесь**, если не можете прочитать `version.json`. Не угадывайте.

## Шаг 1 — Слияние пакетов Unity (`Packages/manifest.json`)

Прочитайте `Packages/manifest.json`. В `dependencies` установите (создайте или
перезапишите только эти два ключа — оставьте все остальные зависимости
нетронутыми), подставив `VERSION` из Шага 0:

```json
"com.alexeyperov.unity-open-mcp-bridge": "https://github.com/AlexeyPerov/unity-open-mcp.git?path=packages/bridge#bridge-v<VERSION>",
"com.alexeyperov.unity-open-mcp-verify": "https://github.com/AlexeyPerov/unity-open-mcp.git?path=packages/verify#verify-v<VERSION>"
```

Пример при `VERSION=0.7.0`:

```json
"com.alexeyperov.unity-open-mcp-bridge": "https://github.com/AlexeyPerov/unity-open-mcp.git?path=packages/bridge#bridge-v0.7.0",
"com.alexeyperov.unity-open-mcp-verify": "https://github.com/AlexeyPerov/unity-open-mcp.git?path=packages/verify#verify-v0.7.0"
```

**Идемпотентные правила:**

- Если ключи отсутствуют → добавьте их.
- Если они уже есть с другим URL/тегом/путём `file:` → **обновите** их до
  привязок выше (публикуемая Git-установка — это путь agent-setup).
- Никогда не удаляйте посторонние пакеты.
- Записывайте корректный JSON (по возможности сохраняйте форматирование;
  предпочитайте минимальный diff).

## Шаг 2 — Настройка MCP-клиента (идемпотентное слияние)

**Предпочитайте локальную для проекта конфигурацию**, когда клиент её
поддерживает. Используйте глобальную конфигурацию, только если человек явно
просит.

Следуйте [Конфигурации MCP-клиента](client-configuration.md): найдите
обнаруженный клиент в таблице, скопируйте его фрагмент, замените
`/absolute/path/to/project` на абсолютный путь из предусловий и замените
каждую npm-версию во фрагменте на **`VERSION` из Шага 0** (общий документ
может отставать от релиза; ваш разрешённый `VERSION` побеждает). Если вы
получили эту процедуру удалённо, загрузите
`https://raw.githubusercontent.com/AlexeyPerov/Unity-Open-MCP/master/docs/setup/client-configuration.md`.

Используйте: `npx -y unity-open-mcp@<VERSION>`.

**Идемпотентные правила слияния, когда `unity-open-mcp` уже существует:**

1. Обновите `command` / `args` (или эквивалент) до `npx -y unity-open-mcp@<VERSION>`.
2. Установите `UNITY_PROJECT_PATH` в абсолютный путь **этого** проекта.
3. Сохраните любые другие env-ключи, уже присутствующие в записи.
4. Оставьте нетронутыми все остальные MCP-серверы / посторонние ключи конфигурации.
5. Создайте родительские каталоги при отсутствии. Если файл не существует,
   создайте его с корректной верхнеуровневой оболочкой для этого клиента.

Не угадывайте форму конфигурации по памяти: общий справочник владеет
клиентскими путями и фрагментами JSON/TOML/CLI. Если OS-глобальный файл
Claude Desktop не удаётся найти, спросите у человека его путь.

## Шаг 3 — Копирование основного скила (только байты)

Установите поставляемый плейбук для **одного** обнаруженного клиента. Файл
большой намеренно; агенты **не** должны его сочинять.

1. При необходимости создайте каталог назначения (таблица ниже).
2. Копируйте оболочечным инструментом — **не** вставляйте и не регенерируйте
   markdown:

   ```bash
   # Удалённо (типичный случай):
   curl -fsSL \
     "https://raw.githubusercontent.com/AlexeyPerov/Unity-Open-MCP/master/skills/unity-open-mcp/SKILL.md" \
     -o "<PROJECT_ROOT>/<client-skill-path>"

   # Вместо этого при локальном чекауте монорепозитория:
   # cp skills/unity-open-mcp/SKILL.md "<PROJECT_ROOT>/<client-skill-path>"
   ```

3. Перезапишите файл, если он уже существует.
4. Убедитесь, что записанный файл нетривиален (порядка сотен строк / десятков
   КБ). Если получился короткий stub — удалите и повторите `curl`/`cp`.
5. **Запрещено во время установки:** переписывать скил, сокращать его,
   дописывать инвентарь проекта вручную или вызывать
   `unity_open_mcp_generate_skill`. Проектная регенерация опциональна позже
   (см. [Скилы](../../skills.md)).

| Семейство клиентов | Путь скила (в корне проекта) |
|---|---|
| Cursor | `.cursor/skills/unity-open-mcp/SKILL.md` |
| Claude (Desktop / Code) | `.claude/skills/unity-open-mcp/SKILL.md` |
| OpenCode | `.opencode/skills/unity-open-mcp/SKILL.md` |
| ZCode / Codex / универсальные агенты | `.agents/skills/unity-open-mcp/SKILL.md` |
| Cline | `.cline/skills/unity-open-mcp/SKILL.md` |
| Gemini | `.gemini/skills/unity-open-mcp/SKILL.md` |
| Kilo Code | `.kilocode/skills/unity-open-mcp/SKILL.md` |
| ZooCode (Roo) | `.roo/skills/unity-open-mcp/SKILL.md` |
| Antigravity | `.agent/skills/unity-open-mcp/SKILL.md` |
| Rider (Junie) | `.junie/skills/unity-open-mcp/SKILL.md` |
| VS Code | `.vscode/skills/unity-open-mcp/SKILL.md` |
| Visual Studio | `.vs/skills/unity-open-mcp/SKILL.md` |
| GitHub Copilot CLI | `.github/skills/unity-open-mcp/SKILL.md` |

Если сопоставление клиента всё ещё неоднозначно после одного вопроса, запишите
**только** путь `.agents/skills/unity-open-mcp/SKILL.md` (папка универсальных
агентов), а не все клиенты.

## Шаг 3б — Отчёт перед передачей (обязательно)

Перед чеклистом USER ACTION выведите точно:

1. `VERSION`, полученный из `version.json`
2. Две строки UPM-зависимостей, которые вы записали
3. Команду запуска MCP, которую вы записали (`npx -y unity-open-mcp@…`)
4. Единственный путь скила и `wc -l` / размер этого файла
5. Подтверждение, что вы **не** выдумали версию и **не** переписали скил

Если какая-либо привязка не равна `VERSION` из Шага 0 — исправьте до продолжения.

## Шаг 4 — Передача USER ACTION (обязательно)

Выведите человеку этот чеклист и **прекратите изменения**, пока он не подтвердит:

1. **Откройте Unity** с **тем же** проектом (`UNITY_PROJECT_PATH`).
2. Дождитесь завершения компиляции скриптов (статус-бар редактора).
3. **Перезапустите ваш MCP / ИИ-клиент**, чтобы он перечитал конфигурацию из Шага 2.
   (Большинство клиентов читает конфигурацию MCP только при запуске.)
4. Опционально: в Unity откройте **Tools → Unity Open MCP Bridge** и убедитесь, что
   статус корректен.

Первый запуск `npx` может занять 10–60 секунд, пока npm скачивает пакет, — это нормально.

## Шаг 5 — Проверка по возможности

После того как человек подтвердит, что Unity открыт и клиент перезапущен:

- Если инструменты Unity Open MCP видны в этой сессии, вызовите
  `unity_open_mcp_capabilities` и/или `unity_open_mcp_ping`.
  - Успех → сообщите, что установка завершена.
  - Неудача → выведите краткий список устранения неполадок ниже; не зацикливайтесь.
- Если инструменты **ещё не видны** → сообщите человеку, что конфигурация/скил/манифест
  на месте и ему по-прежнему нужен перезапуск клиента (или новый агентский чат,
  загружающий MCP). Опционально он может запустить (подставьте `VERSION` и путь
  проекта):

```bash
npx -y unity-open-mcp@<VERSION> wait-for-ready --project /absolute/path/to/project
npx -y unity-open-mcp@<VERSION> run-tool unity_open_mcp_capabilities --project /absolute/path/to/project --json
```

## Краткое устранение неполадок

- **bridge недоступен / соединение отклонено:** Unity должен быть открыт на том же
  абсолютном пути проекта, что и `UNITY_PROJECT_PATH`.
- **`npx` / `node` не найдены:** Node нет в `PATH` — переустановите Node LTS,
  перезапустите ИИ-клиент.
- **Инструменты отсутствуют после правки конфигурации:** перезапустите MCP-клиент.
- **Управляется не тот проект:** `UNITY_PROJECT_PATH` должен быть абсолютным и
  указывать на каталог, содержащий `Assets/`, `Packages/`, `ProjectSettings/`.
- **Установлена неверная / старая версия:** перечитайте `version.json` на Шаге 0
  и перепишите UPM-привязки + MCP `@<VERSION>`; не оставляйте запомненную старую
  привязку.

Подробнее: [Устранение неполадок](../../troubleshooting.md), [Политика диалогов](../../dialog-policy.md).
