# Maintainer versioning and releases

This page owns repository version synchronization, bumps, tags, and release
mechanics. User-facing compatibility and mismatch recovery live in
[Version compatibility](../versioning.md).

## Sources of truth

| Source | Governs |
|---|---|
| `version.json` | Shared npm server + bridge + verify version. |
| `hub/version.json` | Independent Unity Hub Pro app version. |

Every other version string is generated by `scripts/sync-version.mjs`. Do not
hand-edit generated targets.

The shared source updates:

- `mcp-server/package.json`
- bridge and verify `package.json` versions
- the bridge-to-verify dependency pin
- bridge runtime version constants
- setup/client/CI version examples registered in the sync script

The Hub source updates its Tauri config, Cargo package, and npm package version.
Community extension packs and the private root package are independent.

## Sync and drift checks

```bash
node scripts/sync-version.mjs
node scripts/sync-version.mjs --check

node scripts/sync-version.mjs --hub
node scripts/sync-version.mjs --check --hub
```

The version-sync workflow runs both checks for relevant pull requests and
pushes. Fix drift by running the sync command, never by editing one target.
Tool-schema changes also run an advisory token-estimate drift check; regenerate
with `scripts/generate-token-estimates.mjs` when needed.

## Bump or set a version

Shared trio:

```bash
node scripts/sync-version.mjs bump patch   # or minor / major
node scripts/sync-version.mjs set X.Y.Z    # exact version
```

Hub:

```bash
node scripts/sync-version.mjs bump patch --hub
node scripts/sync-version.mjs set X.Y.Z --hub
```

Review and commit the generated changes before creating tags. The Hub app's
maintainer panel can run the same sync/check/bump/set operations, but it does
not commit or tag.

## Release tags

| Tag | Release |
|---|---|
| `vX.Y.Z` | npm MCP server and generated GitHub release notes for the shared trio |
| `bridge-vX.Y.Z` | Bridge UPM git pin |
| `verify-vX.Y.Z` | Verify UPM git pin |
| `hub-vX.Y.Z` | Unity Hub Pro installers and GitHub release |

Create the shared trio's three annotated tags on the same commit:

```bash
node scripts/sync-version.mjs tags X.Y.Z
git push origin vX.Y.Z bridge-vX.Y.Z verify-vX.Y.Z
```

Create a Hub release tag independently:

```bash
node scripts/sync-version.mjs tags X.Y.Z --hub
git push origin hub-vX.Y.Z
```

The `tags` command verifies the version against its source, refuses partial or
existing tag sets, and does not push. Review before running the printed push
command because publishing and GitHub release creation are external,
irreversible actions.

The npm workflow checks the tag and all shared targets, builds the MCP package,
publishes it, and creates a GitHub Release with generated notes. Bridge and
verify have no registry publish step; their tags are the installable UPM refs.
The Hub workflow checks its independent version before building installers and
creating its release.

GitHub Releases are the release-note/changelog owner; do not add a second
hand-maintained changelog.

## Pre-1.0 convention

While the major version is zero, the minor number is the breaking axis:

- `0.5.0` and `0.5.1` are compatible patch drift.
- `0.5.x` and `0.6.x` are incompatible.

After 1.0, standard major-version compatibility applies. A breaking wire/tool
contract change therefore requires a minor bump before 1.0.

## Adding a version surface

1. Add the target to `TRIO_TARGETS` or `HUB_TARGETS` in
   `scripts/sync-version.mjs`.
2. Keep its replacement expression narrow enough to touch only the version
   slot.
3. Add the path to `.github/workflows/version-sync.yml`.
4. Run sync and both relevant check paths.

Prefer reading a runtime source directly. Add a generated target only when that
runtime cannot read the source of truth.

## Runtime compatibility implementation

`mcp-server/src/compat.ts` reads the server package version and compares it with
the bridge's `/ping` version. Pre-1.0 it compares the minor axis; post-1.0 it
compares the major axis. The check is advisory and warns once per server
process. `UNITY_OPEN_MCP_SKIP_VERSION_CHECK=1` suppresses only the warning.

The CLI `status` command reports the same compatibility result in both human
and JSON output.
